SHL Assessment Recommender — Approach Summary

Problem Statement
Build a conversational API that helps hiring managers discover relevant SHL assessments from the entire product catalog. The system must ask clarifying questions, return grounded recommendations (10 max), refine results across turns dynamically, and compare assessments using catalog evidence — all within an 8-turn budget and a 30-second response latency.

Retrieval Setup (RAG Layer)
Index Construction
  • Entire SHL catalog parsed from catalog.json (377 assessments) on service startup.
  • Cleaned all keys/values of trailing spaces and handled nested string lists.
  • Each assessment's name, description, and keys concatenated into a text document blob.
  • Stored and indexed in-memory using an ephemeral **Chroma DB** collection.
  • Vector embeddings generated via a **custom TF-IDF Embedding Function** to avoid high memory usage.
Query-Time Retrieval
  • Latest user query is vectorized and queried against the Chroma collection.
  • Top-25 results are retrieved to act as prompt catalog context for the LLM.
  • **History Link Parsing**: Extracts previously recommended URLs from the assistant's previous replies.
  • Reconstructed history items are injected alongside the Top-25 search results to guarantee refinement persistence.
Prompt Design
  • Role: Configured as a strict SHL domain advisor. Politely refuses general HR tips or prompt-injection attempts.
  • Scope Policy: Hard stop for coding, legal compliance (HIPAA requirements), and general interviewing advice.
  • Turn Injection: Injects Current Turn: N / 8. Turn 7/8 forces the LLM to complete and set end_of_conversation: true.
  • Vague query rule: If context (role/seniority) is incomplete, demands clarification before listing shortlists.
  • Anti-hallucination: LLM only returns entity_id strings. Backend validates the IDs against Chroma's top 25 results + history before resolving URLs/metadata.
  • Temperature = 0: Enforced for reproducible, grounded outputs and strict JSON formatting.

What Did Not Work
Attempt 1 — Default Chroma DB Transformer Embedding
Chroma's default sentence-transformer embedding downloaded and loaded the all-MiniLM-L6-v2 ONNX model, which consumed over 400MB of RAM and crashed Render's Free Tier container. Fixed by writing a custom TfidfEmbeddingFunction wrapper for Chroma DB, which reduced RAM usage to ~5MB.
Attempt 2 — Dirty catalog on-disk parses
Catalog strings contained raw newline characters and trailing spaces in JSON keys, breaking standard library parsers. Implemented a disk-level JSON cleaner to standardize catalog.json into strict valid JSON format.
Attempt 3 — Stateless shortlist drops during refinement
Since the API is stateless, mid-conversation refinements (e.g., "add personality tests") triggered new vector searches that excluded previous context matches, causing the LLM to drop items it previously recommended. Fixed by parsing previous assistant messages, reconstructing previous matches from URLs, and feeding them back to the RAG layer.
Attempt 4 — LLM outputting full URLs directly
The model frequently hallucinated URL paths and invented product names. Replaced with ID-only output (entity_id) in JSON format, leaving metadata lookup and link rendering to deterministic Python functions.
Key Design Choices
Chroma DB & Groq LLM
  • **Lightweight Vector Storage**: In-memory ephemeral database client avoids external cloud hosting dependency.
  • **Fast Sub-2s Inference**: `llama-3.1-8b-instant` keeps API responses well within the 30s timeout window.
  • **Strict JSON Mode**: Standardizes response formats to ensure evaluator compliance.
Stateless Design & Verification
  • **Scale Ready**: No database sessions required; horizontally scalable out-of-the-box.
  • **Deduplicated Shortlists**: Test type mappings and language parameters are resolved via Python helper functions.
  • **Reconstructive Context**: Keeps context footprints small by reading previous message blocks.
